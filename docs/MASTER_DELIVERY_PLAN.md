# ATRI Memories 总体交付规划

目标：先把 ATRI 图片站做成一个基本可用、后续能扩展的小网站；发布、严格交接和完整运维验收放到后续上线阶段处理。后续实现优先保证核心功能能用、结构不要继续堆乱，不再为了当前阶段追求过重的发布级细节。

## 1. 先看当前真实基线

基于当前代码与 Supabase 结构，已确认存在的能力：

- 公开画廊：搜索、筛选、分页加载、精选区、大图查看、单图分享、原图下载。
- 用户能力：注册/登录、重发确认邮件、密码重置、批量上传、编辑/删除自己的图片、个人资料编辑、我的图片筛选、个人收藏。
- 管理后台：运维概览、图片管理、用户管理、举报处理、审计日志、站点设置。
- 权限后端：`memories` / `storage.objects` RLS + `manage-*` / `delete-memory` / `update-memory` / `media-urls` 等 Edge Functions。
- 存储模型：图片文件在 private bucket `atri-images`，前端通过 signed URL 展示和下载。
- 内容治理：公开/下架、精选、举报、审计日志、邀请制发号、全站上传开关、公开注册开关。

当前已经完成的结构收口：

- `gallery / viewer / upload / auth / user / admin` 已进入 `src/features/`。
- `src/styles.css` 已变成分层导入口，具体样式拆到 `src/styles/` 和各 feature CSS。
- `npm run architecture:doctor` 已接入 `npm run verify`，用于阻止业务组件退回 `components/`、样式入口重新堆叠、`App.jsx` 重新膨胀。
- `npm run product:doctor` 已接入 `npm run verify`，用于把用户侧路径、管理侧路径、数据权限边界和交接文档作为本地产品完整性门禁。

当前仍有明确的结构性短板：

- 测试门禁已经覆盖 `build + quality-check + architecture:doctor + product:doctor`，但真实浏览器行为测试和已登录账号路径仍需 `smoke`、手工 QA 与后续部署环境证据补齐。
- 分享预览已具备独立服务端模块雏形，但仍需在真正部署时完成域名、链接策略和回跳配置对齐。
- 运营级能力仍在继续补强，但当前已经具备站点健康、备份/导出、限流/反滥用、运行监控、数据留存策略这些后台模型；后续重点转向更强的上线前验收和真实设备回归。

结论：当前项目已经不是从零开始，而是进入“把已有能力收口为一个基本可用小网站”的阶段。后续重点不该继续发散功能，也不需要现在就把发布级验收做满；先保证核心路径可用、结构边界清楚。

补充约束：

- 核心产品主线优先：公开画廊、用户上传、单图查看、个人中心、图片管理、用户管理、举报处理。
- 扩展运维层保留但降权：监控、备份、留存、上线准备这类能力可以存在，但默认不应压过当前主路径，也不应继续无限膨胀。
- 后台呈现遵循同一原则：常用操作直出，高级运维信息默认折叠或次级展示。

## 2. 当前阶段目标

当前阶段不是做大型产品交付，而是让小网站具备基本功能并保持可扩展。发布前的严格配置和验收留到 Phase 5。

### A. 用户侧基本可用

- 访客能稳定浏览公开内容、检索内容、查看详情、下载、举报、分享。
- 注册用户能上传、维护自己的图片和基本资料。
- 图片详情页覆盖浏览、下载、编辑、删除、举报、分享这些主动作。

### B. 管理侧基本可用

- 管理员可以管理内容、用户、举报、站点设置和审计数据。
- 所有高风险操作都通过服务端权限边界执行。
- 关闭公开注册后，账号发放和用户上传策略有基础入口。

### C. 架构侧不过度堆叠

- 前端按 `app / features / components / hooks / lib` 分层。
- 业务逻辑不继续堆回 `App.jsx`、大组件或单一 hook。
- 媒体访问、权限判断、审计写入、配置读取都有稳定边界。

### D. 发布阶段再补

- 站点具备上线前所需的配置清单、部署策略、分享预览方案和容量监控口径。
- 对异常场景有明确策略：未确认邮箱、举报处理、私有图片访问、下架内容、上传受限、邀请未完成。
- 这些内容作为发布阶段要求，不阻塞当前小网站基础可用。

## 3. 目标架构

### 3.1 前端目标目录

```text
src/
  app/                全局布局、路由状态、弹窗编排、页面级 state
  features/
    gallery/          公开画廊、筛选、分页、精选
    viewer/           大图查看、下载、分享、举报、编辑
    upload/           批量上传、失败重试、上传预览、批次标签
    auth/             登录、注册、确认邮件、重置密码
    user/             个人资料、我的图片、我的统计
    admin/            后台六个页签与后台共享组件
  components/         与业务弱绑定的通用 UI
  hooks/              面向数据源的 hooks
  lib/                Supabase、路由、下载、媒体 URL、标签等工具
  data/               静态预设
```

当前状态说明：

- `app/` 已落地。
- `gallery / viewer / upload / auth / user / admin` 已落到 `features/`。
- 样式已按基础层、app shell、feature 层和响应式层拆分。
- `architecture:doctor` 已把这些边界变成可运行检查，后续继续扩展时优先让新增能力落到对应 feature / lib / hooks，而不是回写到 `App.jsx`、`components/` 或 `styles.css`。

### 3.2 服务端目标边界

保留以下原则：

- RLS 负责第一层权限。
- Edge Function 负责高风险写操作和管理员聚合视图。
- 前端只消费 publishable key，不暴露 service role。
- 图片展示统一通过媒体 URL 适配层，不允许组件直接依赖底层 Storage URL 细节。

### 3.3 数据与权限目标

角色保持固定三层：

- 访客
- 用户
- 管理员

原则固定：

- 不回到口令式管理。
- 不增加“半管理员”“临时权限”这类模糊角色。
- 权限状态由数据库和函数决定，前端按钮显隐只做提示，不做最终授权。

## 4. 模块优先级与冻结顺序

后续开发不再“想到什么做什么”，按下面顺序推进。只有当前阶段达到定义，才进入下一阶段。

### Phase 0：规划冻结

目标：统一规划口径，停止零散加功能。

完成标准：

- 存在一份总体交付规划。
- 架构规划、上线清单、README 都引用同一套路线。
- 后续任务都能明确归属到某个 phase 和模块。

### Phase 1：前端结构收口

目标：把现有功能从“可运行”整理为“可持续扩展”。

必须完成：

- `features/gallery`
- `features/viewer`
- `features/upload`
- `features/auth`
- `features/user`
- 后台 `admin` 继续拆到稳定子模块
- 样式按 feature 或层级分段，不再无限扩张单文件心智负担

完成标准：

- `App.jsx` 只做页面编排。
- 各功能边界明确，不再依赖跨组件隐式状态。
- 代码分割继续保持，构建无异常膨胀。

### Phase 2：内容模型深化

目标：让内容从“只有图片”升级为“可运营内容单元”。

必须完成：

- 标题、描述、标签继续稳定化
- 统一图片状态模型：公开、隐藏、精选、举报中、已处理
- 补充内容排序和更强筛选能力
- 为后续 SEO/详情页能力保留清晰字段边界

可选但不抢跑：

- 标签页
- 专题/系列集
- 首页运营位

当前补充原则：

- 如果继续增强公开内容层，优先把已有标签体系做成真实浏览入口，而不是直接跳到更重的专题/系列编辑系统。

### Phase 3：用户体验闭环

目标：让普通用户不需要后台也能完成完整使用路径。

必须完成：

- 独立的“我的图片/我的资料”体验收口
- 上传结果反馈更完整
- 空状态、失败状态、受限状态更统一
- 邀请制与公开注册两种站点策略都能在前端自然工作

### Phase 4：运营与治理闭环

目标：管理员能真正把站点持续管下去。

必须完成：

- 后台指标口径稳定
- 举报处理闭环稳定
- 用户上传策略稳定
- 审计日志可追溯
- 站点设置覆盖主要运营开关

建议补充：

- 简单健康检查与异常提示
- 数据导出 / 备份策略文档
- 反滥用与频率限制方案
- 注册 / 邀请异常信号
- 邀请发送服务端时间窗口限制

### Phase 5：上线准备

目标：把“能本地用”推进为“可公开部署”。

必须完成：

- 生产域名与 Auth 配置对齐
- 分享预览服务端方案
- 容量与流量监控口径
- 上线前手工验收
- 回归验证脚本保持通过
- 核心烟测脚本保持通过

## 5. 当前之后的固定实现顺序

从现在起，建议严格按下面顺序推进，不再插队：

1. 完成前端 feature 级目录迁移与结构收口。
2. 收口用户侧完整体验：`auth / user / upload / viewer` 的状态与错误处理。
3. 收口公开画廊的数据边界与筛选模型。
4. 补运营侧剩余能力：健康/导出/限流策略。
5. 最后再做真正上线所需的分享预览和生产配置。

这意味着：

- 现在不优先加新奇功能。
- 现在不优先做视觉小改。
- 现在不优先做一次性补丁。
- 先把“可扩展网站”所需的骨架收完整。

## 6. 决策规则

后续任何新需求，都先按这四个问题判断：

1. 它属于哪个模块？
2. 它属于哪个阶段？
3. 它会不会破坏当前边界？
4. 它是补闭环，还是继续发散？

若答案不清楚，就先改规划，不直接改代码。

## 7. 完成定义

当前阶段的“小网站基础可用”满足以下条件即可：

- 用户侧路径完整：浏览、注册/邀请、登录、上传、编辑、删除、下载、举报、分享。
- 管理侧路径完整：概览、内容管理、用户管理、举报治理、审计、站点设置。
- 前后端边界清晰：前端模块化、权限后端化、媒体访问统一化。
- 结构边界可验证：`npm run architecture:doctor` 通过，并作为 `npm run verify` 的一部分。
- 产品主路径可验证：`npm run product:doctor` 通过，能复核用户侧、管理侧、数据权限和交接文档没有在扩展中断线。
- 本地浏览器可验证：`npm run smoke:local` 通过。

发布阶段再要求：

- 部署侧可落地：上线清单完整，分享预览与 Auth 策略有明确方案。
- 验证侧可复核：`npm run smoke`、已登录 smoke 和 `release:preflight -- --strict` 通过。
- 手工验收可交接：`MANUAL_QA_TEMPLATE.md` 和 `npm run qa:init` 生成的记录通过 `qa:doctor`。

## 8. 现在的执行结论

从当前状态出发，后续不应继续零散补功能。正确路线是：

- 先完成前端 feature 级结构收口。
- 再把用户侧体验闭环补完整。
- 再做运营补强与上线准备。

这份文件就是后续实现的主路线图。`SITE_ARCHITECTURE_PLAN.md` 负责模块和边界细节，`LAUNCH_READINESS_CHECKLIST.md` 负责上线前核对项，本文件负责总顺序和完成定义。
